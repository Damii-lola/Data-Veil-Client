DOCUMENTATION · OVERVIEW

What is VeilCipher?

A completely custom character-by-character encryption system that turns any text into a unique hexadecimal string and packages it into a downloadable QR code — all powered by a proprietary 16-step algorithm tied to a secret database of character mappings.

🔐

A Custom Encryption Engine

VeilCipher is a single-page web application that encrypts text using a proprietary 16-step algorithm tied to a secret database of character mappings. The final result is a hexadecimal string and a QR code containing that string.

The entire application is serverless on the frontend and relies on a Node.js backend deployed on Render, which fetches its mapping data from a Supabase PostgreSQL database.

⚡

High-Level Flow

  1. User types a message (e.g. Hello)
  2. Clicks Encrypt & Generate QR
  3. Frontend sends the text to the backend API
  4. Backend encrypts each character individually using the secret mapping table from Supabase, then concatenates the results
  5. The final encrypted string is returned along with every intermediate calculation
  6. Frontend displays the hex, a QR code, and an expandable step-by-step breakdown per character
🗄️

The Database — Character Mapping Tables

The encryption relies on two Supabase tables that together form the secret key of the cipher. Without them, nobody can reproduce the encryption.

cipher_tables
idAuto-increment primary key
table_keyRandom 4-digit number (1000–9999) identifying the full mapping set
created_atTimestamp of creation
character_mappings
characterSingle printable ASCII character (codes 32–126)
mapped_codeRandom unique 9-digit string — digits 1–9 only, no zeros
table_idForeign key → cipher_tables.id

The 9-digit codes are random, unique per table, and never exposed via any API endpoint. This random mapping is the secret key of the cipher.

🔁

Why Is It One-Way?

Several steps lose information irreversibly, making reversal impossible. VeilCipher is essentially a custom cryptographic hash rather than a reversible encryption scheme.

  • Step 6 — Integer division drops the remainder permanently
  • Step 8 — Modulo and division chains discard original numbers; only derived values survive
  • Step 5 — Scramble sums absolute differences — many different 4-digit sets can produce the same sum
📱

QR Code Generation

After the final hex is assembled, the backend uses the qrcode npm library to generate a PNG image of a QR code containing that hex string.

The image is returned as a base64 data URL so the frontend displays it instantly without a separate file request. A download link lets you save the QR as veilcipher_qr.png.

🚀

How to Use It

  1. Scroll up to the Encrypt section
  2. Type any text — letters, numbers, symbols, spaces
  3. Click Encrypt & Generate QR (or press Ctrl+Enter)
  4. The hex appears; the QR code is generated instantly
  5. Expand any character's step block to see the exact math
  6. Copy the hex or download the QR image for later use

The system works fully in the cloud — no local setup needed. The mapping table in Supabase can be regenerated at any time to produce a completely different cipher.

DOCUMENTATION · DEEP DIVE

The 16-Step Algorithm

Every single character of the input text undergoes the exact same sequence of 16 mathematical transformations. The output of each character's journey is concatenated to produce the final encrypted hex string.

WORKED EXAMPLE

All steps below use character A with 9-digit code 517782813 and table key 9148 for illustration purposes.

01

Retrieve from Database

Look up the character in encryptMap (loaded from Supabase at startup). Each printable ASCII character maps to a unique random 9-digit code containing only digits 1–9.

A → 517782813  |  table_key = 9148
02

Split the 9-Digit Code

The code is divided into three segments that will each drive a different part of the calculation pipeline.

First 3: 517  |  Middle 2: 78  |  Last 4: 2813
03

Math on the First 3 Digits

Using digits a=5, b=1, c=7 from the first segment, a series of arithmetic and modulo operations produces a carry value that seeds the rest of the chain.

(a+b)×c = 6×7 = 42  |  a+b+c = 13
mod1 = 42%13 = 3  |  mod2 = 13%42 = 13
Carry = 3 + 13 = 16
04

Multiply with the Middle 2 Digits

The carry value is multiplied by the middle 2-digit integer to amplify the number into a useful range.

carry × midVal = 16 × 78 = 1248
05

Scramble the Last 4 Digits

Two rotation sets are generated from the last 4 digits. Set1 rotates left 3 times; Set2 rotates right 3 times. Pairwise absolute differences are summed into a scramble sum.

last4 = 2813
Set1 (left rotations): [8132, 1328, 3281]
Set2 (right rotations): [3281, 1328, 8132]
Scramble Sum = Σ |Set1ᵢ - Set2ᵢ|
06

Divide and Drop the Decimal

Scramble sum divided by the Step 4 result. Only the integer part is kept — the remainder is permanently discarded. This is one of the lossy steps that makes reversal impossible.

floor(scrambleSum ÷ step4result) = quotient
07

Base-5 Conversion → Decimal Reinterpretation → Multiply

The Step 6 quotient is converted to base-5 notation, then that base-5 string is reinterpreted as a plain decimal number and multiplied by the original first-3 integer.

base5(quotient) = 'X' → treat as decimal → × 517 = product
08

Chain of Calculations

A chain of modulo and division operations uses two divisors — one from the digit-sum of the base-5 string, one from the digit-sum of the last 4 digits. The chain numbers are summed then multiplied to produce chainFinal.

digitSum(base5Str) → divisor1  |  last4Sum → divisor2
chainNums = [divisor1, mod1, mod2, mod3]
chainFinal = sum(chainNums) × divisor1 = chainFinal
09

Weave Two Numbers Together

A custom interleave function weaves digits from chainFinal and div2_8 together in a 1:3 ratio, prefixed by divisor1, producing an expanded numeric string.

weave(start=divisor1, chainFinal, div2_8) = 'weaved string'
10

Triple Base Conversion

The weaved string undergoes three successive base conversions using BigInt arithmetic to handle arbitrary precision throughout.

decimal → base5 → treat as decimal → base7 → treat as decimal → HEX
11

Wrap with Table Signature

The table key is split into two 2-character halves which are prepended and appended to the hex string, embedding a unique table signature in the output.

sig = 9148 → last2=48, first2=91
wrapped = "48" + hex + "91"
12

Swap First 4 and Last 4 Characters

The first and last 4 characters of the wrapped string swap positions, with the middle section remaining intact — a final positional scramble.

"AAAABBBBBBCCCC" → "CCCCBBBBBBAAAA"
13

Read as Hex → Base 10

The swapped string is interpreted as a hexadecimal number and converted to decimal. The result is typically a 30+ digit number, requiring BigInt throughout.

0xSWAPPED → huge decimal (30+ digits)
14

Convert to Base 4

That huge base-10 number is represented in base 4 (digits 0–3). The result is a very long string of 0s, 1s, 2s, and 3s.

huge decimal → base4 string (very long)
15

Treat as Decimal → Base 9

The base-4 string (which only contains digits 0–3, valid as decimal) is reinterpreted as a decimal number, then converted to base 9.

base4 string (as decimal) → base9 string
16

Final: Treat as Decimal → Hex

The base-9 string is parsed as a decimal number and converted to hexadecimal. This is the final output for that character. Hex digits may include 0–9 and A–F.

base9 string (as decimal) → FINAL HEX OUTPUT

The final encrypted string is the concatenation of per-character hex outputs across all input characters.

Frontend — GitHub Pages

A fully static single-page application with no build step and no framework. Pure HTML, CSS, and vanilla JavaScript served directly from the cipher-client GitHub repository.

  • Calls POST /api/encrypt on the Render URL
  • Renders hex output with streaming character animation
  • Displays QR code as a base64 data URL — no extra requests
  • Shows expandable per-character step breakdowns
  • Deployed automatically on every GitHub push
⚙️

Backend — Render

A lightweight Node.js + Express API server deployed on Render's web service tier. Handles all computation-heavy tasks server-side to keep the secret mappings private.

  • Loads the cipher mapping from Supabase at startup
  • Refreshes the mapping cache every 30 minutes
  • Runs the 16-step encryption for every character
  • Generates the QR PNG using the qrcode npm package
  • Returns hex, QR data URL, and full step data as JSON
🗄️

Database — Supabase

A PostgreSQL database hosted on Supabase, holding the secret character mapping tables that form the cipher's key material.

  • cipher_tables — metadata and the 4-digit table key
  • character_mappings — 9-digit code for every ASCII char 32–126
  • Backend connects via the Supabase JS client with a service key stored in environment variables
  • Regenerating the mapping produces an entirely different cipher

Security Model

The security of VeilCipher rests entirely on the secrecy of two values: the character mapping table in Supabase and the table key. Without these, the 16-step algorithm cannot be reproduced even with full knowledge of its structure.

The algorithm itself is documented on this site, but the specific 9-digit codes assigned to each character are secret, randomly generated per-table, and never exposed via any API endpoint. The backend only exposes the encrypted output and intermediate math — never the raw mapping.

The cipher is one-way by design. Multiple lossy steps — integer division, modulo chains, scramble sums — make reversal computationally infeasible.
