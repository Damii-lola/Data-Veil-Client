DOCUMENTATION · OVERVIEW

What is VeilCipher?

A completely custom character-by-character encryption system that turns any text into a unique hexadecimal string and packages it into a downloadable QR code — all powered by a proprietary 16-step algorithm tied to a secret database of character mappings.

01

A Custom Encryption Engine

VeilCipher is a single-page web application that encrypts text using a proprietary 16-step algorithm tied to a secret database of character mappings. The final result is a hexadecimal string and a QR code containing that string.

The entire application is serverless on the frontend and relies on a Node.js backend deployed on Render, which fetches its mapping data from a Supabase PostgreSQL database.

02

High-Level Flow

  1. User types a message
  2. Clicks Encrypt & Generate QR
  3. Frontend sends text to backend API
  4. Backend encrypts each character using the secret mapping
  5. Final encrypted string returned with step data
  6. Frontend displays hex, QR, and step breakdown
03

Why Is It One-Way?

Several steps lose information irreversibly:

STEP 6 Integer division drops the remainder permanently
STEP 8 Modulo chains discard original numbers
STEP 5 Scramble sums absolute differences — many inputs → same sum
04

The Database — Character Mapping Tables

Two Supabase tables form the secret key. Without them, nobody can reproduce the encryption.

cipher_tables
id Auto-increment primary key
table_key Random 4-digit number (1000–9999)
created_at Timestamp
→
character_mappings
character Single printable ASCII char (32–126)
mapped_code Random 9-digit string, digits 1–9 only
table_id Foreign key → cipher_tables.id

The 9-digit codes are random, unique per table, and never exposed via any API endpoint.

DOCUMENTATION · DEEP DIVE

The 16-Step Algorithm

Every character of the input undergoes the exact same sequence of 16 mathematical transformations. The output of each character's journey is concatenated to form the final encrypted string.

Security Model

The security of VeilCipher rests entirely on the secrecy of two values: the character mapping table in Supabase and the table key. Without these, the 16-step algorithm cannot be reproduced even with full knowledge of its structure. The algorithm itself is documented here, but the specific 9-digit codes assigned to each character are secret, randomly generated per-table, and never exposed via any API endpoint. The cipher is one-way by design — multiple lossy steps make reversal computationally infeasible.
